# Common Pitfalls

## Cloudera-Specific Assumptions

Watch for these source assumptions that do not transfer to MRS:

### 1. Impala-Specific Behavior

- `COMPUTE STATS` does not exist in Spark SQL; use `ANALYZE TABLE`
- Impala's `INVALIDATE METADATA` -> `MSCK REPAIR TABLE` or `REFRESH TABLE`
- Impala query hints (`/* +SHUFFLE */`) are not supported
- Impala's low-latency MPP model differs from Spark's throughput model

### 2. CDH-Specific Configuration

- Cloudera Manager configuration does not transfer to MRS Manager
- CDH parcels and packages are not compatible with MRS
- Cloudera-specific JARs must be replaced with MRS equivalents
- CDH-specific `hdfs-site.xml` and `core-site.xml` properties may not apply

### 3. Hive ACID Tables

- Cloudera Hive ACID (transactional) tables cannot be directly migrated as external tables
- Export ACID tables to Parquet first, then create as external tables on MRS
- If transactional semantics are required, use MRS Hive managed tables with ACID

### 4. Kudu Integration

- Cloudera Kudu has no direct equivalent on MRS
- Options: HBase, CloudTable, or Parquet with periodic compaction
- Kudu-specific SQL syntax must be rewritten

### 5. Oozie Workflows

- Oozie is not available on MRS
- Migrate to: MRS JobGateway, DolphinScheduler, or Apache Airflow
- Oozie coordinator jobs -> scheduled Spark jobs

## Huawei Cloud / MRS Operational Pitfalls

### 1. Parquet Dictionary Encoding (Most Common)

**Problem:** MRS Hive 3.1's Parquet reader does not support dictionary-encoded Parquet files generated by Spark 3.x.

**Error:**
```
UnsupportedOperationException: org.apache.parquet.column.values.dictionary.PlainValuesDictionary$PlainIntegerDictionary
```

**Solutions:**
1. Use `spark-sql` instead of `beeline` for querying Spark-generated Parquet (recommended)
2. Re-write Parquet without dictionary encoding for Hive compatibility
3. Use Spark to create Hive-compatible managed tables

### 2. OBS Path Issues

**Problem:** OBS paths use `obs://` scheme, not `hdfs://`.

Common mistakes:
- Using `hdfs://` paths where `obs://` is intended
- Incorrect bucket names or region endpoints
- OBS 404 errors due to path encoding (`%2F` for `/`)

**Fix:**
- Always verify OBS paths with `obsutil ls`
- Ensure the OBS endpoint matches the bucket region
- The `%2F` encoding in OBS paths is normal; Spark handles it transparently

### 3. HiveServer2 Connection

**Problem:** HiveServer2 binds to the Master node's internal IP, not `localhost`.

**Error:**
```
Connection refused (Connection refused)
```

**Fix:**
- Use the Master node's internal IP (e.g., `192.168.0.145:10000`)
- Find the IP from `netstat -tlnp | grep 10000`
- Or use ZooKeeper-based discovery (may require Kerberos on security clusters)

### 4. File Permissions

**Problem:** Files created as `root` cannot be read by `omm` user.

**Fix:**
- Always set `chmod 644` for SQL files before running as `omm`
- Use `su - omm` for all Hadoop/Hive/Spark operations

### 5. beeline File Access

**Problem:** `beeline -f <file>` fails with "Permission denied" when the file is owned by root.

**Fix:**
```bash
chmod 644 /tmp/script.sql
su - omm -c 'source /opt/Bigdata/client/bigdata_env && beeline -u "jdbc:hive2://<ip>:10000" -f /tmp/script.sql'
```

### 6. EIP Binding

**Problem:** The V2 EIP SDK `update_publicip` API may fail with "Invalid param".

**Fix:** Use the V3 EIP SDK instead:
```python
from huaweicloudsdkeip.v3.model.update_publicip_option import UpdatePublicipOption
body = UpdatePublicipOption(
    associate_instance_type="PORT",
    associate_instance_id=<port_id>
)
```

### 7. Security Group Rules

**Problem:** The VPC V2 SDK `create_security_group_rule` may fail with "invalid params".

**Fix:** Use the VPC V3 SDK with `multiport` instead of `port_range_min/max`:
```python
from huaweicloudsdkvpc.v3.model.create_security_group_rule_option import CreateSecurityGroupRuleOption
body = CreateSecurityGroupRuleOption(
    security_group_id=<sg_id>,
    direction="ingress",
    protocol="tcp",
    multiport="22",
    remote_ip_prefix="0.0.0.0/0",
    ethertype="IPv4",
    action="allow",
    priority=1
)
```

### 8. MRS Version Constraints

**Problem:** Some MRS versions are marked as test-only and cannot be created.

**Fix:**
- Always query available versions for the target region
- Prefer LTS versions (e.g., MRS 3.5.0-LTS)
- If a version is rejected, try the next available LTS version

### 9. MRS Node Spec Constraints

**Problem:** MRS requires `.linux.bigdata` suffix for node specs.

**Fix:**
- Use `c6.4xlarge.4.linux.bigdata` (not `c6.4xlarge.4`)
- Disk sizes have service-enforced minimums (typically 600 GB)
- Core node minimum is 3 (not 1)
- Master node minimum is 2 (not 1)

### 10. Spark SQL Output Parsing

**Problem:** Spark SQL terminal output mixes progress bars with result data, making automated parsing difficult.

**Fix:**
- Save results to HDFS using `INSERT OVERWRITE DIRECTORY`
- Read from HDFS using `hdfs dfs -cat`
- Or redirect output to a file and parse carefully

## Authorization Pitfalls

### OBS Access from MRS

- **AK/SK method:** Configure `obsutil` or Spark `fs.obs.access.key`/`fs.obs.secret.key`
- **Agency method (recommended for production):** Create an MRS agency with OBS permissions
- **Temporary credentials:** Use STS tokens for short-lived access

### Kerberos

- MRS supports Kerberos but it changes the access pattern significantly
- All Hadoop commands require `kinit` before execution
- beeline requires Kerberos authentication
- For demo/non-production clusters, use non-secure mode (safe_mode=0)

### Ranger

- Cloudera Sentry policies need to be converted to Ranger policies
- Ranger is included in MRS 3.5.0-LTS by default
- Use MRS Manager UI to configure Ranger policies
