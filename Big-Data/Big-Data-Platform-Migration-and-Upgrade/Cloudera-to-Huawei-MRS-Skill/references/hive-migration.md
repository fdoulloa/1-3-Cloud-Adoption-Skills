# Hive Migration

## Hive Metastore Migration

### Source: Cloudera Hive Metastore

Cloudera uses a MySQL/PostgreSQL-backed Hive Metastore. MRS also uses an internal Metastore (DBService).

**Do not attempt to copy the Metastore database directly.** Instead, extract DDL and recreate tables.

### DDL Extraction

```bash
# On Cloudera, extract DDL for all tables
beeline -u "jdbc:hive2://<host>:10000" -e "SHOW CREATE TABLE finance_dw.fact_transaction;"
```

### DDL Conversion Rules

| Cloudera Hive Feature | MRS Hive Equivalent |
| --- | --- |
| `STORED AS PARQUET` | Same |
| `STORED AS ORC` | Same |
| `STORED AS TEXTFILE` | Same |
| `LOCATION 'hdfs://...'` | `LOCATION 'obs://...'` or `LOCATION 'hdfs://...'` |
| `PARTITIONED BY (col type)` | Same |
| `TBLPROPERTIES` | Most properties transfer; skip Cloudera-specific ones |
| `SERDE` | Most SerDes transfer; verify on MRS |
| Hive ACID tables | Not directly supported as external tables; export to Parquet first |

## External Table Pattern

Prefer external tables over managed tables for migrated data:

```sql
-- External table on OBS (recommended for migration)
CREATE EXTERNAL TABLE finance_dw.fact_transaction (
  ...
)
PARTITIONED BY (txn_date_key int)
STORED AS PARQUET
LOCATION 'obs://<bucket>/<project>/finance_dw/fact_transaction';

-- External table on HDFS (alternative)
CREATE EXTERNAL TABLE finance_dw.fact_transaction (
  ...
)
PARTITIONED BY (txn_date_key int)
STORED AS PARQUET
LOCATION 'hdfs:///user/hive/warehouse/finance_dw.db/fact_transaction';
```

**Why external tables:**
- Data persists independently of the table definition
- Safer for migration: dropping the table does not delete data
- Can point to OBS for cloud-native storage
- Easy to re-create or re-point

## Partition Recovery

After creating partitioned external tables, recover partitions:

```sql
-- Add all partitions found in the location
MSCK REPAIR TABLE finance_dw.fact_transaction;

-- Or add specific partitions manually
ALTER TABLE finance_dw.fact_transaction ADD PARTITION (txn_date_key=20250401);
ALTER TABLE finance_dw.fact_transaction ADD PARTITION (txn_date_key=20250402);
```

## Hive ACID Tables

Cloudera Hive ACID (transactional) tables are not supported as external tables on MRS. Migration path:

1. Export ACID table to Parquet:

```sql
-- On Cloudera
INSERT OVERWRITE DIRECTORY '/export/<table>'
STORED AS PARQUET
SELECT * FROM <acid_table>;
```

2. Create as external Parquet table on MRS (non-transactional)

3. If transactional semantics are required on MRS, use Hive 3.x ACID on MRS managed tables

## Hive View Migration

Views typically transfer directly, but watch for:

- Impala-specific functions (see impala-migration.md)
- Hard-coded HDFS paths in view definitions
- References to tables that have not been migrated yet
- `LATERAL VIEW` with table-generating functions (usually compatible)

## Parquet Compatibility Issue

**Critical:** MRS Hive 3.1 cannot read dictionary-encoded Parquet files generated by Spark 3.x.

Error:
```
UnsupportedOperationException: org.apache.parquet.column.values.dictionary.PlainValuesDictionary$PlainIntegerDictionary
```

Solutions (in order of preference):

1. **Use Spark SQL instead of Hive for queries** (recommended)
2. **Re-write Parquet without dictionary encoding:**

```sql
-- In Spark SQL
INSERT OVERWRITE DIRECTORY '/tmp/hive_compatible/<table>'
USING PARQUET
OPTIONS ('parquet.enable.dictionary' = 'false')
SELECT * FROM finance_dw.<table>;
```

3. **Use Spark to create Hive-compatible managed tables:**

```sql
-- In Spark SQL
CREATE TABLE finance_dw.<table>_hive STORED AS PARQUET AS
SELECT * FROM finance_dw.<table>;
```
