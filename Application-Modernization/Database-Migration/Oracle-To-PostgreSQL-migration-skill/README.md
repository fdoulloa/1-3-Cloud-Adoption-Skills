# Skill: Oracle → PostgreSQL 17 Migration

Converts SQL code written in Oracle dialect to PostgreSQL 17 within an application project. Operates iteratively: scans the source code, applies standard transformations, verifies semantic gaps in the DDL, and detects runtime bugs not visible in pattern scanning.

Language and framework agnostic.

---

## When to use this skill

Invoke when the user wants to:

- Migrate Oracle queries to PostgreSQL in an existing project (PHP, Java, Python, TypeScript, Ruby or other)
- Audit a codebase for residual Oracle patterns
- Verify that a DDL generated by an automated tool (UGO, AWS SCT, ora2pg) has no semantic gaps
- Diagnose post-migration runtime errors related to dialect differences

Typical trigger phrases: _"migrate queries to PostgreSQL"_, _"convert Oracle SQL"_, _"check the code for Oracle patterns"_, _"I'm getting errors after migrating to PG"_.

---

## Argument

```
/skill oracle-to-postgres [scope]
```

`[scope]` — target connection or schema. Examples:
- `"default oracle connection"`
- `"sales schema"`
- `"all connections"`

If omitted, Phase 1 asks interactively.

---

## Execution phases

| Phase | Description |
|-------|-------------|
| **1 — Scope** | Confirms which connection/schema to migrate, which ones stay on Oracle, and which tool generated the PG DDL |
| **2 — Scan** | Recursive `grep` of ~20 Oracle patterns in source code; classifies each hit as in-scope / out-of-scope / comment / dead code |
| **3 — Transformations** | Applies equivalence tables: dates, temporal arithmetic, data types, NVL/DECODE, ROWNUM, sequences, CONNECT BY, legacy outer joins, DDL types |
| **4 — Strategy** | Decides `replace_all` vs targeted edit based on ambiguity and connection mixing |
| **5 — Validation** | Re-scan to confirm zero Oracle residuals in scope |
| **6 — Identifier case** | Compares PG DDL against code; fixes uppercase tables and columns; verifies `AS NAME` aliases |
| **7 — Post-tool DDL gaps** | Detects orphan sequences (known UGO issue), Oracle triggers not ported, missing defaults; delivers idempotent SQL patch script |
| **8 — Runtime gotchas** | Checks: empty string vs NULL, Oracle driver APIs missing in PG, outdated sequence START after restore, uppercase VARCHAR values as identifiers, silent try/catch, ORM identifier quoting |

---

## Covered transformations

**Date/time functions**
`SYSDATE` · `TRUNC(SYSDATE)` · `ADD_MONTHS` · `MONTHS_BETWEEN` · `TO_DATE` · `TO_CHAR` with `RRRR` formats

**Conditionals and nullability**
`NVL` · `NVL2` · `DECODE`

**Pagination**
`ROWNUM` as simple filter, as subquery wrapper, and as COUNT guard

**Sequences**
`seq.NEXTVAL` · `seq.CURRVAL` · `CREATE SEQUENCE NOCACHE`

**Series generation**
`CONNECT BY LEVEL` · `FROM dual`

**DDL types**
`VARCHAR2` · `NUMBER` · `CLOB` · `BLOB` · `DATE` (with time) · `TIMESTAMP WITH TIME ZONE`

**Joins**
Legacy `table(+)` syntax → `LEFT/RIGHT JOIN`

## Prerequisites

- Read/write access to the project source code
- PostgreSQL dump file or live schema (required for Phases 6 and 7)
- Original Oracle dump available (only needed when truncated sequences have no clear heuristic match)

---

## Notes on DDL migration tools

Phase 7 adjusts its behavior based on the tool declared in Phase 1:

| Tool | Main known gap |
|------|----------------|
| **UGO (Huawei Cloud)** | Does not link sequences as column PK `DEFAULT`; truncates sequence names to 30 chars |
| **AWS SCT** | May omit audit triggers; review automatic timestamps |
| **ora2pg** | Usually handles defaults well, but may fail with custom domain types |
| **Manual conversion** | No predefined gaps; review §7.2 regardless |

---

## Deliverables per phase

- **Phase 2**: classified hit inventory
- **Phase 5**: clean scan confirmation (or list of justified exceptions)
- **Phase 6**: report of corrected identifiers
- **Phase 7**: idempotent SQL script (`init-sequences.sql` or equivalent Flyway/Liquibase) + list of Oracle triggers not ported with justification
- **Phase 8**: list of detected gotchas with fix applied or recommended

---

## Limitations

- Does not migrate stored procedures, packages or PL/SQL functions — only SQL embedded in application code
- Does not directly modify the PostgreSQL DDL; Phase 7 patches are delivered as scripts to be applied manually
- The Phase 2 scan may produce false positives in complex comments or strings containing Oracle keywords; classify edge cases manually
